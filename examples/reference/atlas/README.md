# Atlas reference (blusys)

Connects an ESP32 / ESP32-C3 / ESP32-S3 to an **Atlas** deployment,
composed entirely from blusys framework primitives:

| Capability                     | Role                                           |
|--------------------------------|------------------------------------------------|
| `blusys::connectivity_capability` | Wi-Fi + SNTP                                |
| `blusys::mqtt_capability`         | MQTTS to EMQX with per-device token         |
| `blusys::ota_capability`          | OTA download + flash (dynamic URL)          |
| `atlas_capability` (this example) | Atlas wire conventions: topic layout, retained-state shape, OTA announcement → `ota.request_update(url)` |

The Atlas adapter holds no transport or OTA plumbing of its own —
everything heavy sits in framework capabilities. That's what makes new
products cheap to build: they compose the same primitives, and the
product-specific code stays thin.

## Layout

- `main/core/` — state, actions, reducer, op-state machine
- `main/platform/` — `atlas_capability` + capability wiring + `on_tick`
- `main/include/atlas_config.h` — generated by `tools/provision.py` (gitignored)
- `main/include/ca_cert.h` — generated by `tools/provision.py` (gitignored)
- `tools/provision.py` — register device with Atlas, render config, flash

## Provisioning and flashing

Use `tools/provision.py`. It logs into Atlas, registers the device,
fetches the EMQX CA from `GET /infra/ca.crt`, renders
`main/include/atlas_config.h` + `main/include/ca_cert.h`, and invokes
`idf.py`.

```sh
# source ESP-IDF first (`. $IDF_PATH/export.sh`)
python3 tools/provision.py \
    --atlas-url http://atlas.example.com:3000 \
    --admin-password 'super-secret' \
    --name living-room-sensor \
    --target esp32s3 \
    --wifi-ssid my-ssid \
    --wifi-pass 'my-pass' \
    --mqtt-host broker.example.com
```

The device should connect to the broker within ~10s of boot and appear
online in the Atlas dashboard.

Add `--skip-flash` to build without flashing, or `--port /dev/ttyUSB0`
to pin the serial port. `--admin-token` accepts a pre-issued JWT if
you'd rather not pass `--admin-password`.

## Event protocol

Reserved integration event IDs — posted by `atlas_capability` into the
blusys runtime and delivered to the reducer via `app_spec::map_event`
(as `capability_event_tag::integration_passthrough`):

| ID       | Symbol                            | Meaning                                |
|----------|-----------------------------------|----------------------------------------|
| `0x0901` | `atlas_event::broker_connected`   | CONNACK received, subscriptions active |
| `0x0902` | `atlas_event::broker_disconnected`| Session lost — reconnect pending       |
| `0x0910` | `atlas_event::command_received`   | `atlas/{id}/down/command`              |
| `0x0911` | `atlas_event::state_desired`      | `atlas/{id}/down/state`                |
| `0x0912` | `atlas_event::ota_announcement`   | `atlas/{id}/down/ota`                  |

Connect-attempt failures surface via the canonical
`capability_event_tag::mqtt_error` on the underlying `mqtt_capability` —
the Atlas adapter does not re-emit them under a custom ID.

IDs live inside the reserved **product-custom** band `0x0900–0x09FF`
(see `blusys/framework/capabilities/capability.hpp`).

## Build without provisioning

You can build/flash manually once `main/include/atlas_config.h` has been
rendered at least once (or copied from the PlatformIO reference):

```sh
idf.py set-target esp32s3
idf.py build flash monitor
```

## Wire-protocol validation (no hardware)

`tools/atlas_blusys_sim.py` mirrors what `atlas_capability` emits on the
wire: same topics, same JSON shapes, same LWT, same command-ack event. Run
it from a machine that can reach the Atlas EMQX broker to confirm the
server side (auth, ACL, state storage, command routing) accepts what the
firmware will send.

```sh
pip install --user paho-mqtt

# Register a device via the Atlas admin API, capture {id, mqttToken}, then:
python3 tools/atlas_blusys_sim.py \
    --device-id "$DEVICE_ID" \
    --mqtt-token "$MQTT_TOKEN" \
    --broker localhost --port 8883 \
    --ca-cert /path/to/atlas/infra/emqx/certs/ca.crt
```

The sim relaxes `VERIFY_X509_STRICT` because the dev CA lacks the
`keyUsage` extension that Python's `ssl` module now enforces; ESP-IDF
mbedtls has no such requirement, so the on-device firmware is unaffected.
